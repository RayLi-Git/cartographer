# 設計

> 一個作品集的價值不在「做了什麼」，而在「為什麼這樣做」。本文件記錄 Cartographer 的設計理念與關鍵決策：一份外流的硬體 PRD，怎麼變成一個可複用的 PRD 起草 skill 的藍本。
> 涵蓋邊界與未來方向見 [SCOPE](./SCOPE.md)；本文只談「為什麼」。

---

## 設計理念

Cartographer 的原料不是憑空生出來的——它起點是一份外流的 **Apple AirPods Pro 硬體 PRD**。那份文件骨架紀律極佳：每條需求原子化、編號、帶優先級（P1–P10）、可量測（FOB <$75、漏音 <10dB）、persona 驅動、誠實列開放問題，附名詞表與競品分析。第一個關鍵洞察就定了全案的調：

> **這些紀律與「硬體」無關，正是好軟體 PRD 該有的 DNA。** 所以移植骨架，再修掉藍本做錯的地方。

由此落出五個核心信念：

1. **不是需求，除非它「原子＋編號＋優先級＋可驗收＋可追溯」。**
2. **不可量測的形容詞是毒** — 快/好/友善/無縫 都要翻成數字。
3. **非功能需求與資安要被強迫面對，不可跳過。**
4. **誠實列開放問題 > 假裝想清楚。**
5. **每條需求都要能追回它的來源。**

架構與同門一致：**指路牌常駐、細節按需。** `SKILL.md` 保持精簡（其 description 也受 ~1024 字元上限約束）以保證觸發，真正的訪談細則按模組從 `references/` 按需載入，絕不吃爆 context。

| 層 | 角色 | 比喻 |
|---|---|---|
| `SKILL.md` | 常駐指路牌 + 觸發協議 | 門柱招牌（精簡、按需載 references） |
| `references/` | 15 個主題模組的詳盡訪談指引 | 圖書館書庫（需要時才去翻） |
| `scripts/` + `templates/` | 可跑的 lint + 空白/填好 PRD 範本 | 工具箱（機械化強制的實體形態） |

## 關鍵決策

### 決策 1：以 AirPods Pro PRD 為藍本

- **問題**：一份「好 PRD」到底長什麼樣？抽象建議（「要清楚、要完整」）誰也教不會，具體又有紀律的範本才行。
- **選擇**：採用外流的 AirPods Pro 硬體 PRD 當骨架藍本，移植它的優點——原子化編號需求、優先級、可量測驗收、persona 驅動的故事、誠實的開放問題、名詞表、競品分析。
- **取捨**：硬體 PRD 帶有硬體專屬雜訊（BOM 成本、聲學漏音）要濾掉。但「結構」可乾淨轉移，而真實範本永遠勝過憑空編造的建議。

### 決策 2：用 6 個資深視角審視，再修掉藍本的 8 個缺點

- **問題**：藍本雖好仍有破綻——而這些破綻剛好也是軟體 PRD 最常見的坑。照單全收就會繼承這些洞。
- **選擇**：用 6 個資深視角審視藍本，逐一修掉 8 個具體缺點，每個對應到一個模組。

  | # | 藍本缺點 | Cartographer 的修正 |
  |---|---|---|
  | 1 | 目標把假設當事實（「will expand 15%」） | §02 強制分開 目標/假設/事實，並要量測機制 |
  | 2 | 風險混進 Open Questions | §03 獨立「假設·約束·風險」，風險登記含機率×衝擊×緩解 |
  | 3 | 收健康資料卻零隱私 | §08 資安升格獨立模組，深掛 Sentinel |
  | 4 | 功能需求多半沒驗收條件 | §06 每條一律強制 AC，prd_lint 把關 |
  | 5 | 只寫 happy path | §06 強制補負向/邊界/狀態（空·載入·錯誤·離線） |
  | 6 | 無需求依賴圖 | §12 加需求依賴排序，餵 Compass 施工順序 |
  | 7 | 有零件沒連線（需求不知來源） | 可追溯矩陣織全程，§14 產出 |
  | 8 | 不可量測的爛句 | 收進各模組當「反例庫」教材 |

- **取捨**：比一張光禿模板多了模組、多了紀律。但每個修正都堵住一個洞，否則它會在工程開工後變成返工。

### 決策 3：15 模組，而非一個大 SKILL.md

- **問題**：所有訪談指引該塞進一個檔案，還是拆開？
- **選擇**：照官方三層漸進揭露——metadata 永遠載入、`SKILL.md` 觸發時載入（精簡 <2000 字）、`references/` 按需載入。15 模組讓輕量任務只碰幾節，不吃爆 context。
- **取捨**：檔案變多、要導覽。換來的是：一個小改版（4 節）不會被拖進整套資安/NFR 機器，而觸發保持可靠。

### 決策 4：P0–P3，而非藍本的 P1–P10

- **問題**：藍本的十級優先級雖精準，但溝通成本高，也不是軟體團隊講話的方式。
- **選擇**：用軟體界標準 P0(阻斷)/P1/P2/P3——溝通成本低、團隊通用。§13 名詞表註明對應（P0≈P10），保留可量化精神。
- **取捨**：顆粒度變粗。但優先級是溝通工具，一把全團隊讀法一致的尺，勝過一把沒人共享的細尺。

### 決策 5：資安要「獨立模組 + 深掛 Sentinel」

- **問題**：藍本最大漏洞就在資安/隱私，且這是軟體 PRD 最常被跳過的一段。把它折進泛泛的「NFR」一行，保證被略過。
- **選擇**：把 §08 升格為獨立模組，掛上 Sentinel 的資安思考習慣與紅旗清單——讓「寫死金鑰/不驗證輸入/給全權限」在 *PRD* 階段、還沒有 code 之前就被堵。
- **取捨**：資安段變重，🟢 輕量任務可跳、但 🔴 含金流/PII/權限的任務不可跳。這個不對稱是刻意的：上線後才發現的隱私缺口，代價遠高於訪談時間。

### 決策 6：可追溯矩陣當交棒介面

- **問題**：一份每條都格式良好的 PRD，仍可能「有零件沒連線」——需求沒有可追溯來源、目標沒有任何需求服務它。
- **選擇**：把可追溯織進全程，並在 §14 產出矩陣——每條需求標來源/目標/AC/里程碑。這是超越藍本之處，也是交棒 Compass 最乾淨的介面：孤兒需求與落空目標一眼可見。
- **取捨**：起草時多了帳要記。但這正是 Compass 反向審計要吃的，成本付一次、下游重用。

### 決策 7：逐節停下，不一次生成

- **問題**：一次生成 PRD 會產出「看似完整、實則臆測」的內容——模型用貌似合理的猜測填補使用者從沒確認的空缺。
- **選擇**：逐節訪談、每節停下等確認。PRD 品質來自逼問細節的人機對話；每次停下確保每節有真實的使用者輸入。
- **取捨**：比一次成稿慢。但一份塞滿未確認假設的快稿不是合約，是負債。

### 決策 8：機械化優先——靠 exit code 不靠紀律

- **問題**：「記得每條需求都編號、補驗收條件、別用模糊形容詞」這種建議，會在期限壓力下崩壞，跟所有靠人自律的規則一樣。
- **選擇**：`prd_lint.py` 把「缺編號/缺優先級/缺 AC/缺來源/不可量測形容詞」變成可阻斷、帶非零 exit code 的檢查——呼應 Compass「靠 exit code 不靠紀律」的信念。一次真實壓測再把它迭代到 v2：移除散文層形容詞警告（精準度太低——連官方範例都中；形容詞改成只在「需求行內」判定為阻斷），並新增兩個結構檢查（重複編號＝阻斷、懸空 `依賴:` ＝警告）。
- **取捨**：lint 擋得住「格式錯的需求」，卻擋不住「根本沒編號的需求」——無編號 PRD 會得到假 PASS。這個盲點正是審查模式 skill 層存在的理由：人驅動的訪談補上機械閘擋不住的部分。

## 它刻意不做的事

這些是刻意畫的邊界，不是缺陷。完整理由在 [SCOPE](./SCOPE.md)；簡版：

- **它不實作 PRD。** Cartographer 停在一份可交棒的規格；照規格施工、不偏航是 [Compass](https://github.com/RayLi-Git/compass) 的事。
- **它不管你怎麼想。** 根因診斷與資安思考習慣是 [Sentinel](https://github.com/RayLi-Git/sentinel) 的範疇——§08 引用它，而不重教。
- **它不寫硬體 PRD。** 藍本雖是硬體，但 Cartographer 為軟體行為與狀態調校——機構/電性/製造規格不在範圍。
- **它不假裝一次成稿就是合約。** 逐節停下就是重點；跳過去「直接生成」就毀了整個設計。

---

> Cartographer 不是萬靈丹——它是**特定情境下精準的工具**。它探索「如何把『將模糊想法逼成可驗收 PRD』的紀律，編碼成 AI 寫程式的搭檔」。它的同門探索其餘部分：[Compass](https://github.com/RayLi-Git/compass) 照圖施工、[Sentinel](https://github.com/RayLi-Git/sentinel) 管你怎麼想、[Lookout](https://github.com/RayLi-Git/lookout) 在桅杆瞭望。
