# Cartographer 設計決策與取捨

> 記錄「為什麼這樣設計，不那樣」。讓未來維護者（含 AI）理解取捨脈絡。

---

## 1. 為什麼以 AirPods Pro PRD 為藍本

一份外流的 Apple AirPods Pro 硬體 PRD，骨架紀律極佳：每條需求原子化、編號、帶優先級（P1–P10）、可量測（FOB <$75、漏音 <10dB）、persona 驅動、誠實列開放問題、附名詞表與競品分析。**這些紀律與「硬體」無關，正是好軟體 PRD 該有的 DNA**，所以移植。

## 2. 用 6 個資深視角審視藍本後，修掉它的 8 個缺點

藍本雖好仍有破綻（也是軟體 PRD 常見坑），逐一修正：

| # | 藍本缺點 | Cartographer 的修正 |
|---|---|---|
| 1 | 目標把假設當事實（「will expand 15%」） | §02 強制分開 目標/假設/事實，並要量測機制 |
| 2 | 風險混進 Open Questions | §03 獨立「假設·約束·風險」，風險登記含機率×衝擊×緩解 |
| 3 | 收健康資料卻零隱私 | §08 資安升格獨立模組，深掛 Sentinel |
| 4 | 功能需求多半沒驗收條件 | §06 每條一律強制 AC，prd_lint 把關 |
| 5 | 只寫 happy path | §06 強制補負向/邊界/狀態（空·載入·錯誤·離線） |
| 6 | 無需求依賴圖 | §12 加需求依賴排序，餵 compass 施工順序 |
| 7 | 有零件沒連線（需求不知來源） | 可追溯矩陣織全程，§14 產出 |
| 8 | 不可量測的爛句（1.6/2.11/3.5） | 收進各模組當「反例庫」教材 |

## 3. 關鍵取捨

### 為什麼 15 模組、而非一個大 SKILL.md
官方三層漸進揭露：metadata 永遠載入、SKILL.md 觸發時載入（精簡 <2000 字）、references 按需載入。15 模組讓輕量任務只碰幾節，不吃爆 context。

### 為什麼 P0–P3，而非藍本的 P1–P10
軟體界 P0(阻斷)/P1/P2/P3 溝通成本低、團隊通用。§13 名詞表註明對應（P0≈P10），保留可量化精神。

### 為什麼資安要「獨立模組 + 深掛 Sentinel」
藍本最大漏洞就在資安/隱私，且這是軟體 PRD 最常被跳過的一段。掛 Sentinel 的 11 條資安習慣與紅旗清單，讓「寫死金鑰/不驗證輸入/給全權限」在 PRD 階段就被堵。

### 為什麼要可追溯矩陣
這是超越藍本之處，也是交棒 compass 的最佳介面：每條需求標來源/目標/AC，孤兒需求與落空目標一眼可見。

### 為什麼逐節停下、不一次生成
PRD 品質來自人機對話逼問細節。一次生成易產生看似完整、實則臆測的內容；逐節停下確保每節有使用者輸入與確認。

### 為什麼角色語氣（中英皆是）
與既有 compass-en / sentinel-en 一致（"I'm Compass" 式第一人稱），品牌體感統一。frontmatter description 仍用第三人稱關鍵詞包以利觸發。

## 4. 機械化優先（靠 exit code 不靠紀律）
`prd_lint.py` 把「缺編號/缺優先級/缺 AC/缺來源/不可量測形容詞」變成可阻斷的檢查，呼應 compass「靠 exit code 不靠紀律」的信念。

### prd_lint v2（依真實壓測回饋迭代）
一次端到端壓測發現兩個問題：散文裡的形容詞（如「快速」）會誤報、連官方範例都中；且 lint 漏查結構性錯誤。v2 修正：
- **移除**散文層形容詞警告（精準度太低）——形容詞只在「需求行內」判定為阻斷。
- **新增**兩個結構檢查：重複編號（阻斷）、懸空依賴（`依賴:` 指向本文件不存在的 FR/NFR → 警告），用 exit code 取代人工孤兒檢查。
> 教訓：lint 擋得住「格式錯的需求」，擋不住「根本沒編號的需求」（無編號 PRD 會得到假 PASS）——這正是審查模式 skill 層存在的理由。
